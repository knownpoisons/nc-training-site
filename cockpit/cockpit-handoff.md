---

## What this is

A Slack-native pipeline operating system for a solo founder running B2B outreach for a $50k training programme. It generates each day's outreach actions, writes every draft, tracks the pipeline, and keeps the founder honest against a $700k annual revenue target.

**The user is a creative, not a coder, and not a natural salesperson.** Every design decision optimises for: zero app-switching, zero blank-page moments, zero "what should I do now" decisions. The interface is a single Slack channel. If a feature requires the user to open something other than Slack or their email client, it's wrong.

**The system drafts. The human sends.** Nothing is ever sent automatically to a prospect or subscriber. Non-negotiable in v1.

## The daily loop (what Jem experiences)

```
07:00  Morning Brief posts to #cockpit.
       Max 8 actions. Each action = prospect name, touch number,
       full draft with [PERSONALISE] gap marked, context line
       ("replied to touch 2 on Tuesday — this is the ask").
       One stakes line at top: pipeline vs target, plain numbers.

Day    Jem replies in channel as he works:
       "done 1 2"            → logs touches sent
       "skip 3"              → reschedules +2 days, logs skip
       "replied 4"           → moves prospect to REPLIED, asks
                                Jem to paste the reply, drafts response
       "add [name, role, co]" → creates prospect, assigns sequence
       "rewrite 2 shorter"   → redrafts via Claude, reposts
       "call booked 5"       → stage → CALL, prompts for date
       Anything else in plain English → conversational handler

14:00  If zero interactions today: ONE nudge. Stakes attached,
       computed, not motivational. Never a second nudge.

FRI    Scoreboard: sent / replies / calls / closes this week,
       per engine, per track (A: agencies, B: brand teams),
       streak, day-90 gate countdown, honesty line.

MONTHLY (1st Tuesday) Newsletter draft posted to #cockpit in
       Beehiiv-paste-ready format + subject options. Alert:
       "Review, paste, schedule for Wednesday 9am ET."
```

## Key functions — build these, nothing else

### F1 — Morning Brief (cron, 07:00 Jem local time)
Queries touches due today, orders by value (replies to handle first, then warm, then cold touches), caps at 8, generates any drafts not yet written, posts as one threaded Slack message. If nothing is due: posts the lightest possible line ("Nothing due. One thing worth doing: [single suggestion from backlog]"). Never posts an empty brief, never posts more than 8 actions — overflow rolls to tomorrow via the load balancer.

### F2 — Draft Engine
Every draft generated by Claude API with: the touch template, prospect record (name, role, company, notes, prior touches, any replies), VOICE_BRIEF, PLAYBOOK, PROOF. Output rules enforced in code, not just prompt:
- **Banned-words linter** (hard fail, regenerate): synergy, leverage, unlock, journey, empower, robust, ecosystem, holistic, disrupt, game-changing, best-in-class, circle back, touch base, deep dive.
- **Stat guard** (hard fail): any dollar figure or percentage in a draft must appear verbatim in PROOF.md. Otherwise regenerate without it.
- Under 120 words for outreach touches. One CTA. First line contains the [PERSONALISE] marker — the system never fakes personal knowledge it doesn't have.
- British spelling.

### F3 — Cadence Engine
Touch schedule per new prospect: day 1, 4, 12, 21. Rules:
- Weekend touches shift to Monday.
- **Daily load cap: 8 touches.** Overflow shifts ±1 day, earliest-added prospect wins.
- Reply at any point halts the sequence and moves prospect to REPLIED.
- Skip pushes that touch +2 days; two consecutive skips on same prospect → flag in Friday scoreboard.
- After touch 4 with no reply: prospect → DORMANT, resurfaces in 90 days.
- Volume governor: 6 new prospects/week default. After 3 consecutive weeks of ≥80% brief completion, prompt Jem: "Streak earned. Raise to 8/week?" Never raise without his yes. (Validated load: 6/wk ≈ 8 touches/day steady state ≈ 30 min; 8/wk ≈ 10/day ≈ 40 min. Never exceed 8/wk in v1.)

### F4 — Slack Command Layer
Parse plain-language replies in #cockpit. Structured intents: done, skip, snooze, add, rewrite, replied, call booked, closed won/lost, show pipeline, pause [prospect], settings. Anything unmatched routes to F5. Every state change gets a ✅ reaction or one-line confirmation — never silent.

### F5 — Conversational Handler
Unmatched messages go to Claude API with: system context (what the Cockpit is, current pipeline state, Jem's profile), PLAYBOOK, conversation thread. This is how Jem thinks out loud ("this prospect went cold after pricing, what's the move?"). It can propose actions ("want me to schedule a breakup touch for Friday?") but only executes on confirmation.

### F6 — Friday Scoreboard + Honesty Report
Computed, never generated-only: touches sent vs due, reply rate, calls booked, per engine (partner / outbound / alumni / list) and per track (A: agencies / B: brand teams). Day-90 gate countdown with current track leader. The honesty line uses real conversion data once ≥20 touches exist; before that, uses plan assumptions and says so. Format: numbers first, one sentence of read, no cheerleading.

### F7 — Newsletter Builder (monthly cron)
Prompts Jem three days prior: "What's this month's one insight? Reply with rough notes or a voice-note transcript." Builds the email from his notes + VOICE_BRIEF + PROOF, formats as clean HTML/rich text for Beehiiv paste, posts with 3 subject line options. If Jem doesn't supply notes, it drafts from the month's Cockpit activity (calls booked, patterns seen) and flags it as machine-sourced for heavier review.

### F8 — The Nudge
14:00 cron: if no channel interaction since Morning Brief, post one line: current stakes number + the single highest-value action from the brief. Once. Never twice. If two consecutive days pass untouched, the Friday scoreboard names it plainly; no daily escalation. A guilt machine gets muted — this must never become one.

## Build phases with validation gates

**Claude Code: do not advance a phase until its gate passes. Post gate results as test output Jem can see.**

### Phase 1 — Data + Cadence (no Slack, no AI)
Build: schema, cadence engine, load balancer, cron skeleton.
**Gate 1 (automated tests):**
- Unit tests: weekend shift, +2 skip, reply halt, dormancy at touch 4, load cap 8 with overflow ordering.
- **14-day simulation:** seed 12 fake prospects across 2 weeks at 6/week; assert every touch fires on the correct day, no day exceeds 8, no touch lost. Print the day-by-day schedule for eyeball review.

### Phase 2 — Slack loop (fake drafts)
Build: bot, Morning Brief, command parser, confirmations, Nudge.
**Gate 2 (round-trip test, run in a test channel):**
- Brief posts at cron time with seeded data.
- Each command (done/skip/add/replied/rewrite/pause) round-trips: message → state change in DB → confirmation in channel. Assert DB state after each.
- Nudge fires at 14:00 only when interaction count = 0; assert it never fires twice.

### Phase 3 — Draft Engine + Conversational Handler
Build: Claude API integration, linters, stat guard, F5.
**Gate 3:**
- Generate 20 drafts across all touch types with seeded prospects. Run banned-words linter and stat guard across all 20 — zero failures allowed in final output (regeneration loop must catch them).
- Adversarial test: put a fake stat ("$9M saved") in a prospect note; assert it cannot appear in a draft (only PROOF.md numbers pass).
- Voice spot-check: post 5 drafts to the test channel for Jem's read. Jem approves or the templates/prompts get revised. **This gate requires Jem.**

### Phase 4 — Scoreboard + Newsletter + polish
Build: F6, F7, settings, volume governor.
**Gate 4:**
- Scoreboard computes correctly from seeded event history (hand-calculate one week, assert match).
- Newsletter builder produces paste-ready output from sample notes; Jem pastes into a real Beehiiv draft once to confirm formatting survives.

### Phase 5 — Dry-run week (the real gate)
One full week, real cadence, 6 REAL prospects, Jem operating it daily. Nothing counts as shipped until Jem has sent real touches from real briefs for 5 consecutive weekdays. End of week: 30-minute review — what got ignored, what felt heavy, what was missing. Fix before scaling volume.

## Failure modes to design against

1. **The guilt machine.** Over-nagging → channel muted → system dead. One nudge, ever, per day. Friday is the only reckoning.
2. **Draft genericness.** If drafts read like sales automation, Jem stops trusting them and the blank-page problem returns with extra steps. The PLAYBOOK and [PERSONALISE] markers are the defence. Never fake personalisation.
3. **Stat leakage.** One unverified number in one email to one procurement-minded prospect kills a $50k deal. The stat guard is a hard gate, not a prompt suggestion.
4. **Volume fantasy.** The governor exists because a tracker calibrated to aspiration gets abandoned by week three. 6/week until earned.
5. **Silent state drift.** Every command gets visible confirmation. If Jem can't trust that "done 3" registered, he stops replying.

## ENGINE ZERO — Lead sourcing, enrichment, and scoring

The layer that fills the queue the cadence engine drains. **Critical rule: Engine Zero feeds a reviewed queue, never the live sequence. The 6/week volume governor is untouched.** Jem promotes leads from queue to sequence via the weekly digest (F11). No source, however warm, bypasses this.

### The five sources

```
SOURCE               WARMTH      INTAKE            APPROACH LANE
Scorecard completer  HOT         Live webhook      Scorecard follow-up
                                 (form → API)      (2-touch)
Inbox correspondent  WARM        One-time mine     Personal re-open
(getcontent@ Gmail)              of history        (1-touch, hand-written
                                                   gap is large)
Community list       WARM        CSV import        "Never launched"
(200+, big names)                (sources folder)  opener (2-touch)
LinkedIn engager     SLIGHTLY    Export import     Engager sequence
(comments/DMs)       WARM        (sources folder)  (3-touch)
Newsletter sub       BROADCAST   Beehiiv webhook   NEWSLETTER ONLY.
(2,500)              ONLY        (new subs) +      Promoted to pipeline
                                 one-time export   only on signal (reply,
                                                   scorecard, click)
```

**Compliance rule, hard-coded:** newsletter subscribers consented to a newsletter, not 1:1 sequences. A sub enters the pipeline only via a signal event (replied to a broadcast, completed the Scorecard, or explicit Jem promotion with a reason logged). The importer must not mass-create IN_SEQUENCE prospects from the sub list. Same principle for the community list: personal, honest, individual outreach — never a merge-field blast.

### F9 — Intake
- **Scorecard webhook:** modify the nc-training-site repo — Scorecard form POSTs name, email, company, answers, and score to `POST /api/leads/scorecard` (authed). Their answers ARE the personalisation — the follow-up draft references their actual result.
- **Beehiiv new-subscriber webhook** → lead record, Tier C default, scored up by Pass 1.
- **CSV/export importers** for: community spreadsheet, LinkedIn interactions export, Beehiiv sub export, inbox-mine output. Importers must be built against the REAL files (headers vary) — see Gate 0.
- **Inbox mine:** one-time, run as a Claude session with Gmail access (not built into the app): extract correspondents with 2+ way threads from the last 24 months, classify relationship, output CSV for the importer. Live Gmail OAuth sync is v2.
- **Dedupe on ingest:** merge on email, fallback name+company. Multi-source hits stack — each additional source adds to warmth score. A community-list + LinkedIn-engager + scorecard lead is Tier A by definition.

### F10 — Enrichment & scoring (tiered — never deep-research everyone)
- **Pass 1 (free, all leads):** parse what the data already holds — corporate vs personal email domain, role/title if present, company, engagement recency, source warmth. Compute score.
- **Score = warmth (source, stacked) × fit (role seniority; org type: agency / brand team / studio) × recency.** Thresholds set tiers: A (deep research + personal approach), B (light research, queued), C (broadcast lane).
- **Pass 2 (Tier A only):** enrichment job, max 10/day: web search on person + company → two-line dossier (who they are, what they're doing now, why they'd care) + suggested opener angle + source citations. **Dossier guard (hard rule): every claim in a dossier carries its source URL; no found fact, no claim. Invented biography in an opener is a killed deal.**
- Enrichment writes `tier`, `score`, `dossier`, `opener_angle` to the lead record.

### F11 — Weekly Lead Digest (Monday, posts before the Morning Brief)
Top 10 queued leads by score: name, company, source(s), tier, dossier line, proposed lane. Jem replies "promote 1 3 4" / "bin 2" / "hold 5". Promoted leads enter their source-matched sequence within the weekly volume cap. If Jem promotes more than the cap, the system schedules the overflow across coming weeks and says so.

## THE DECK — web review surface (Phase 6)

**Rule that survives all design ambition: Slack runs the day. The Deck is where Jem reviews, not where he operates.** If a daily action requires opening The Deck, the design has failed. Build order is strict: The Deck starts only after Gate 5 (dry-run week) passes.

### Scope — four screens, one panel
1. **Pipeline** — the prospect table as a bento of paper cards + a sortable list. Inline stage moves, source badges, tier chips.
2. **Lead Queue** — the weekly digest in visual form: dossier cards, promote/bin/hold actions mirroring the Slack commands.
3. **Scoreboard** — the Friday numbers as a living dashboard: per-engine, per-track, day-90 gate countdown as the hero insight card.
4. **Settings** — volume governor, brief hour, templates editor.
5. **Assistant panel** (persistent, right rail on desktop, sheet on mobile) — chat UI wired to F5, same brain as the Slack conversational handler. Shows referenced records as chips.

Auth: single user, magic link. No team features.

### Design language — "Cobalt Paper"

The aesthetic: engineered calm. Warm paper-white field, disciplined data cards with hairline borders, and one place where the light gets in — frosted cobalt gradient glass, stacked like cards in a hand. Serif titles against grotesk data. Gradients where the system *speaks* (insights, AI); hard crisp edges where the system *reports* (data). Never mixed.

#### Colour tokens
```
--ink            #0A0A0B     primary text, black UI blocks
--canvas         #F6F6F4     app background (warm off-white, never pure)
--paper          #FFFFFF     data cards
--hairline       #E9E9E7     1px card borders, dividers
--text-2         #8A8A85     secondary text, labels
--cobalt         #1437E8     primary accent, actions, active states
--cobalt-deep    #0A24B8     hover/pressed
--cobalt-glow    #E9EDFF     tint fills, selected rows
--glass-a        #2E5BFF     gradient start (insight cards)
--glass-b        #A9BCFF     gradient end
--signal-warm    #0A0A0B     markers inside progress bars (black, not red)
```
No green anywhere. No red/amber alert colours — state is communicated in ink, cobalt, and weight. Success is a filled cobalt check; problems are plain black text that says what's wrong.

#### Typography
```
Display   Instrument Serif (Google Fonts) — screen titles ("Pipeline"),
          insight headlines inside glass cards, and ONLY those. Set
          large (clamp 40–72px), regular weight, tight leading (0.95),
          slight negative tracking. The serif is the personality; use
          it with restraint or it becomes wallpaper.
UI/Body   Geist Sans — everything else. Labels 12px/medium/tracking
          +0.02em, body 14–15px.
Data      Geist Sans with font-variant-numeric: tabular-nums.
          Stat numerals oversized (40–64px), weight 500, tracking
          -0.03em, with the % or unit as a superscript at 40% size,
          raised — the screenshot detail. Numbers use comma decimal
          styling only if locale says so; default to standard.
```
Pairing logic: the serif carries editorial confidence, the grotesk carries the data. They never do each other's job — no serif numerals, no grotesk display titles.

#### Card system — two species, strictly separated
**Glass cards (the system speaks):** cobalt gradient (135°, glass-a → glass-b), backdrop-blur(40px), 24px radius, white text with ink headline. Rendered as a stacked deck — active card front, two rear cards offset 12/24px, scaled 0.96/0.92, slightly rotated (±1.5°), blurred. Used for: the stakes/insight headline, day-90 gate hero, assistant identity. Maximum one glass stack per screen.
**Paper cards (the system reports):** --paper fill, 1px --hairline border, 20px radius, 24px padding, shadow none at rest. Icon chip + 12px label top-left, oversized numeral, then the barcode bar. All data lives here.
**Hard-edge counterpoint:** inside the soft cards, small elements go sharp — 4px radius on tags, source badges, and the black active tile in the mobile bottom nav; square stroke-caps in the barcode bars. The tension between 24px cards and 4px internals is deliberate. Radius scale is exactly {4, 12, 24, 999} — nothing in between.

#### Signature element — the barcode bar
The segmented tick progress bar from the reference: 1.5px vertical strokes, 3px gap, 16px tall. Unfilled strokes --hairline; filled portion --cobalt; current-position marker a 6px-wide --ink block. Used for every progress/ratio in the product (sequence progress, weekly completion, track comparison). This is the one element The Deck is remembered by — it appears nowhere as a generic rounded progress bar.

#### Micro-interactions (all transform/opacity only — 60fps is a requirement, not a hope)
```
Page load     Cards enter staggered 40ms apart: y 12px→0, opacity
              0→1, 400ms cubic-bezier(0.22,1,0.36,1). Title serif
              enters first, alone.
Numerals      Count up from 0 over 600ms ease-out on first view per
              session. Never on re-render.
Barcode bars  Strokes fill left→right, 20ms/stroke, on scroll into
              view (IntersectionObserver, once).
Glass stack   Swipe/drag carousel, spring physics (framer-motion,
              stiffness 260, damping 24). Rear cards parallax at 0.5x.
Card hover    translateY(-2px) + shadow 0 8px 24px rgba(10,10,11,.06),
              180ms. Arrow chips (↗) rotate to → on hover.
Charts        Line draw-in via stroke-dashoffset, 800ms, once.
Row actions   Promote: row slides right 8px with cobalt-glow flash,
              then settles into new position (layout animation).
              Bin: height collapse, 240ms.
Data loading  Skeleton shimmer in card shapes. Never spinners.
Reduced motion  prefers-reduced-motion: all transforms disabled,
              opacity-only fades, count-ups render final values.
```
One orchestrated moment (the load sequence) over scattered effects. If an animation doesn't clarify state or hierarchy, cut it.

#### Responsive behaviour
```
≥1280px   12-col bento. Assistant panel persistent right rail (360px).
1024      8-col, assistant collapses to floating button + sheet.
768       Cards stack 2-up, glass stack full-width.
≤480      The phone reference layout: single column, glass hero,
          floating pill bottom-nav (paper, blur, black 12px-radius
          active tile), all primary actions in thumb reach.
```
Touch targets ≥44px. The mobile view is not a shrunk desktop — it's the phone screenshot's pattern rebuilt.

#### Build notes
Next.js App Router, Tailwind with the tokens above as CSS variables (no hardcoded hex in components), framer-motion, custom SVG for barcode bars and charts (recharts only if styled to tokens). Copy in the UI follows the voice brief: plain verbs, sentence case, no filler — "Promote", "Bin", "3 due today", never "Supercharge your pipeline!". Empty states direct action ("No leads queued. The Monday digest fills this."), errors say what happened and the fix.

### Gate 6 — The Deck ships only if:
- Lighthouse performance ≥ 90 on the Pipeline screen with 200 seeded prospects.
- 60fps scroll and carousel on a mid-range phone (profile, don't assume).
- Keyboard: every action reachable, visible focus states in cobalt.
